复习Spring

1、框架 就是一个开发好的程序，重点在于怎么使用，记住用法之后再了解原理
2、使用原因是在开发过程中，让操作变得简单
3、Spring核心 IOC（DI）、AOP
(1) IOC控制反转，平常的new对象叫主动控制，控制反转就是把对象的管理交给Spring的容器
(2) DI：依赖注入 进一步简化控制反转的操作
4、引入依赖
5、操作applicationContext.xml文件，写入要生成的的对象的bean
6.Spring容器：Spring 容器是Spring 框架的核心。 容器将创建对象，把它们连接在一起，配置它们，并管理他们的整个生命周期从创建到销毁。 Spring 容器使用依赖注入（DI）来管理组成一个应用程序的组件

spring 注解方式管理bean对象

(1) 注解：只要带@***** 这就是注解，利用反射的原理，实现框架的功能
##注解
Java代码中出现以@开头的内容，都是注解

  1. @Overide表示程序员告诉Java下面的方法是重写方法的方法，如果不是java会报错

  2. @SuppressWarings(“unused”)表示程序员告诉java下边的变量定义不要 报警高

  3. @Deprecated 表示告诉java下面的方法是过期的，在调用这个方法时会有标注

  4. @Configuration注解标记表示程序员告诉Java，这个类是配置Spring容器内容的类

  5. @Bean注解标记表示程序要告诉Java下面这个方法返回的对象，要放入spring容器

利用注解实现框架功能

Spring的组件扫描

现在Spring的问题

当我们需要一个类的对象作为组件注入时，我们必须在config类中使用@Bean声明一个方法，返回这个类的对象，如果需要注入的对象很多，那么代码量就会提升，对于那么简单的直接实例化就可以返回的注入，我们可以使用组件扫描自动进行

组件扫描也是注解

组件扫描也是一个注解，它一般便携在@Configuration注解的下面
@ComponentScan(basePackages=”****”)
Component：组件
Scan：扫描
basePackages：基础包
从指定的基础包开始扫描这个包中的所有类，使用组件注解注入组件对象，

实现组件扫描的步骤

1.在Congfig类中添加ComponentScan注解，并指定扫描的包
2.在想注入到Spring容器中的类上加注解@Component
1)工作原理：在Spring容器（ACAC）实例化时，会自动扫描ComponentScan注解指定的包
2)指定的包和这个包中所有子包的类都会被扫描，所有类前标记有Component注解或功能相同的注解都会自动注入到Spring容器
3.创建测试类测试

小结：组件注入Spring容器的两种方式

Spring的Bean管理

Bean管理

1.Spring容器中可能有很多我们注入的组件，这些组件在从容器中提取出来并使用时，会有一些细节上的区别和区分
2.Bean管理就是设置Spring容器中组件的细节的差别

Bean的作用域设置

1.注入Spring容器中的组件在获得时，是不是同一个对象，如果是，我们称之为单例对象，如果不是，我们称之为原型独享

Bean的生命周期

  1. 从对象的创建到销毁的过程
  2. 何时创建？对象一般实在实例化对象时创建Spring容器中默认情况下再初始化ACAC对象时，Spring容器中所有的对象也创建了
  3. 何时销毁？
  1. 一般在GC回收时销毁对象（手动销毁如close（）方法
  2. Spring容器中默认情况下是在ACAC对象被销毁时，Spring容器中的所有对象也就销毁了

Bean对象生命周期管理

Spring容器中的对象何时创建，何时销毁

默认情况下

默认情况下Spring容器实例化时，容器中的对象也创建，Spring容器销毁时容器中的对象也销毁

非默认情况下

  1. 当注入的对象是Scope为”prototype”类型
    a. Spring容器实例化时，容器中prototype的对象不会实例化
    b. Spring容器销毁时，容器中prototype对象不会销毁，原因是如果Spring负责维护过多数量的对象，难免会有内存浪费甚至造成溢出
  2. 当我们设置单例对象为懒惰初始化（延迟加载）在类前要开@Lazy
    Spring容器实例化时，容器中被设置为@Lazy的类不会实例化在程序运行过程中
    如果使用getBean方法获得这个对象，才实例化

使用@Import配置多个配置文件

(1) 我们在配置Spring容器中的内容只能通过Config类，如果一个项目组多人同时对一个Config继续宁配置，难免会有冲突
所以此时在原Config文件中引入@Import(day02.cof.Config2.class)其它的Config就可以引入其它的Config内容

Spring DI （依赖注入）

依赖注入是什么

1.就是将组件（对象）保存到Spring容器中管理的操作，依赖注入就是程序运行过程中，业务需要依赖的组件也注入到Spring中，并且可以在Spring内部形成依赖关系

2.Bean实现依赖注入

  1. 注入青龙偃月刀，创建一个类（DrangonBlade）这个类有一个返回武器名称的方法
  2. 注入关羽，还能一个类（GuanYu）这个类有一个DrangonBlade属性
  3. 创建Config类，使用@Bean注入DrangonBlade和GuanYu注入GuanYu时的方法需要一个DrangonBlade参数，在方法中，将这个参数复制为GuanYu对象的DrangonBlade属性
    4） 测试类中，获得关羽的Bean可以直接调用关羽的方法，
  4. 注意：如果Spring容器中注入了一个DranonBlade的对象，那么注入GuanYu的方法的参数明随意即可，但是如果Spring容器中只注入多个对象，那么GuanYu的方法的参数名必须和其中一个方法名一致

@Autowired(自动装配)实现依赖注入（DI）

@autowired要做的工作其实就是初始化我们类的成员变量

  1. 创建DragonBlade和GuanYu类代码一致，但是要加上@Component注解
  2. 创建Config类，无需编写任何@Bean只需要扫描组件包即可，在GuanYu类中的DragonBlade属性生添加@AutoWired注解，标识自动装配需要的对象@AutoWired注解同时支持编写在属性的Set方法上，使用相对灵活

IOC/DI的解耦

耦合性

  1. 程序中一个组件（对象）依赖另一个组件时，就会出现耦合性问题
  2. （主对象（关羽）依赖对象(龙刀)）如果主对象中声明的时依赖对象的具体类型，就是紧耦合，
  3. 如果主对象中声明的时依赖对象的抽象类型，就是松耦合
  4. 我们追求的时松耦合，为什么呢？因为松耦合能最大限度的实现易维护易扩展的目的

@Autowired自动装配规则

  1. 如果依赖对象的类型在Spring容器中没有匹配对象，运行报错（异常）
  2. 如果依赖对象的类型在Spring容器中只有唯一匹配对象，运行正常
  3. 如果依赖对象的类型在Spring额容器中有多个匹配对象，且没有匹配对象的id，运行报错
  4. 如果依赖对象的类型在Spring容器中有多个匹配对象，且有匹配对象的id，运行正常，匹配对象的ID的方法有两种
  1. 主对象声明的依赖对象的属性名使用Spring容器中的组件的id
  2. 主对象生命的依赖对象的属性名上使用@Qualifier注解在（）编写组件id
